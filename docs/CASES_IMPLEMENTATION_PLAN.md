# План внедрения кейсов: Studify, Hub, Mars

## Цель

Показать клиенту с КП готовые примеры: три полноценных страницы кейсов с продающим контентом и спокойными анимациями. Кейсы закрывают возражение «покажите, что уже делали» и дают повод запросить сайт после КП.

---

## 1. Копирайтинг и позиционирование

**Тон:** B2B, без воды. Фокус на задаче клиента, решении и измеримом результате. Короткие предложения, цифры где уместно.

**Структура каждого кейса:**
- **Один абзац** — кто клиент, чем занимается, в чём была боль.
- **Задача** — что хотели достичь (рост заявок, экономия времени, автоматизация).
- **Решение** — что мы сделали (бот, CRM, дашборд, интеграции); без перечисления технологий ради технологий.
- **Результат** — 2–4 конкретных пункта (проценты, сроки, метрики или качественные сдвиги).
- **Стек/инструменты** — одна строка (AmoCRM, Telegram Bot API, Next.js и т.д.) для доверия технических специалистов у клиента.

**Три кейса:**
1. **Studify** — консалтинг по обучению за рубежом. Боль: ручная обработка заявок, потеря лидов. Решение: квалификация через бота, CRM, автоматические напоминания. Результат: рост заявок, сокращение времени ответа.
2. **Hub** — коворкинг. Боль: бронирование переговорок вручную, путаница с оплатой. Решение: бот/форма бронирования, календарь, уведомления. Результат: заполняемость, меньше ручной работы.
3. **Mars** — учебный центр. Боль: запись на курсы, напоминания, проверка домашек. Решение: личный кабинет, бот с расписанием и напоминаниями. Результат: удержание, автоматизация рутины.

---

## 2. Структура страниц и роутинг

- **Главная:** секция «Кейсы» — три карточки (изображение/плейсхолдер, название, краткий результат, ссылка «Подробнее»). Ссылка «Все кейсы» ведёт на `/keisy`.
- **Страница списка** `/keisy`: те же три карточки в виде сетки, заголовок, подзаголовок. Для клиента с КП можно прислать ссылку «наши кейсы».
- **Страница кейса** `/keisy/[slug]`: slug = `studify` | `coworking-hub` | `mars`. Контент: hero кейса (название, ниша, один слоган), блок «Задача», «Решение», «Результат», блок «Стек», CTA «Обсудить проект».

Данные кейсов хранятся в `src/data/cases.ts`: один массив с полями для списка и для полной страницы. `generateStaticParams` для трёх slug.

## 3. Анимации (GSAP)

**Принцип:** заметные, но не отвлекающие. Без постоянного движения и «игрушечных» эффектов.

**Главная и страница списка:**
- Заголовок секции: появление при скролле (opacity + y), один раз.
- Карточки: появление по очереди (stagger 0.1–0.15s), лёгкий y снизу, ease power3.out. Hover: лёгкое поднятие (translateY -4px) и усиление тени через CSS, без GSAP.
- Кнопка «Все кейсы»: лёгкое появление после карточек.

**Страница кейса:**
- Hero: заголовок и подзаголовок — короткая последовательность (badge → заголовок → подзаголовок), без посимвольной анимации.
- Блоки «Задача», «Решение», «Результат», «Стек»: ScrollTrigger, start при входе в viewport (например top 85%), анимация opacity 0→1 и y 24→0, duration 0.6–0.7, один раз.
- Цифры в блоке «Результат»: при появлении блока можно опционально запустить лёгкий счётчик (от 0 до значения) или оставить статичными для простоты.
- CTA внизу: тот же приём reveal при скролле.

**Технически:**
- ScrollTrigger один раз зарегистрировать (например в layout или в одном месте).
- Все анимации отменять в cleanup (ctx.revert() в useEffect).
- Проверять `prefers-reduced-motion: reduce` — при нём отключать или сильно упрощать анимации (только fade, без движения).

## 4. Визуал и компоненты

- Палитра: только переменные из globals.css (--background, --primary, --card, --foreground, --muted-foreground). Никаких хардкодных цветов из Hero (голубой/фиолетовый) в кейсах.
- Карточка: фон --card, бордер тонкий, скругление по дизайн-системе. Плейсхолдер изображения — градиент или нейтральная заглушка с иконкой/буквой.
- Страница кейса: тот же layout (Header + main + Footer). Hero кейса — компактный блок с названием и нишей. Блоки с заголовком H2 и текстом; список результатов — маркеры или иконки «галочка».

---

## 5. Порядок работ

1. Файл данных `src/data/cases.ts` с тремя кейсами и типами.
2. Компонент `CaseCard` (карточка с ссылкой, описание, метка ниши).
3. Компонент `CasesSection` для главной: заголовок, три `CaseCard`, кнопка «Все кейсы», GSAP для заголовка и карточек.
4. Страница `app/keisy/page.tsx`: заголовок, подзаголовок, три `CaseCard`.
5. Страница `app/keisy/[slug]/page.tsx`: выбор кейса по slug, hero, блоки контента, CTA; GSAP для блоков.
6. Обновить главную `app/page.tsx`: заменить заглушку «Кейсы (в разработке)» на `CasesSection`.
7. Header и Footer: ссылки «Кейсы» ведут на `/keisy`; логотип на `/`.

После этого у клиента есть три готовых кейса и две точки входа: с главной и отдельная страница «Кейсы» для рассылки после КП.
